Go CLIs Course  ·  Lesson 1

Test-drive rndm: red → green → refactor

Build the first version of rndm — a stand-in for bash's echo $RANDOM — by writing the test before the code. One loop of the TDD cycle, one working binary.

Your win for this lesson By the end you will have (1) driven a Go function into existence from a failing test first, (2) learned the one honest way to test something random — assert a property, not a value, and (3) shipped a rndm binary that prints a number to stdout and exits 0, the way clig.dev says a good CLI should.

1. What we're building, and why start here

In bash, echo $RANDOM prints a pseudo-random integer between 0 and 32767. That's our whole spec for v0 of rndm: print one random integer in [0, 32767] and exit. No flags, no Cobra — we add those only when a real need shows up. Small on purpose: it lets us put all our attention on the discipline that carries the rest of the course — test-first development.

Why test-first at all You've written Go tests before — after the code. TDD flips the order: the test comes first and defines what "done" means before any implementation exists. The payoff is that every line of production code you write is there to satisfy a concrete, executable requirement — and you get a regression suite for free. We'll make the rhythm explicit.
RED
Write a small test. Run it. See it fail.
→
GREEN
Write the least code to pass.

2. Set up the module

Make a directory for the project and initialise a Go module. The module path is just an identity for your code — use your own GitHub path: 

mkdir rndm && cd rndm
go mod init github.com/<you>/rndm

That writes a go.mod:

module github.com/<you>/rndm

go 1.26.1

A command-line program lives in package main. We'll keep two files: rndm.go (the code) and rndm_test.go (the tests). Go recognises any file ending in _test.go as tests — they compile only under go test, never into your binary.

3. RED — write the failing test first

Here's the crux of testing randomness: you can't assert the output equals a specific number — it's random by design. If you knew the answer, it wouldn't be random. So instead you assert a property that must hold for every possible draw. Our property: the result is always within [0, 32767]. We check it across many draws so a stray out-of-range value can't hide.

Create rndm_test.go — note the function Number doesn't exist yet. That's the point.

// rndm_test.go
package main

import "testing"

func TestNumberIsInBashRandomRange(t *testing.T) {
	for i := 0; i < 10000; i++ {
		n := Number()
		if n < 0 || n > 32767 {
			t.Fatalf("Number() = %d; want a value in [0, 32767]", n)
		}
	}
}
TestXxx
Go only runs funcs whose name starts with Test and that take *testing.T.
t.Fatalf
Report the failure and stop the test immediately. Right choice here: once one draw is out of range, the property is broken — no need to keep looping.
the loop
10,000 draws. Property tests trade certainty for confidence — more draws, more coverage of the random space.

Now run it. It should not compile — because Number is undefined:

go test ./...
# github.com/<you>/rndm [github.com/<you>/rndm.test]
./rndm_test.go:7:8: undefined: Number
FAIL	github.com/<you>/rndm [build failed]
A build error is a valid RED Newcomers sometimes think "red" means a test ran and reported a wrong value. Not necessarily — a compile error counts. The test names something that doesn't exist yet, so the code won't build. That failing feedback is exactly what drives the next step: make it exist.

Why does the test assert a range instead of comparing to an exact expected number?

Test the property, not the value. A random function has no single correct output to compare against. What you can pin down is an invariant true of every possible result — here, that it lands in [0, 32767]. Later we'll see the other approach: inject a seeded source so the sequence becomes fixed and exactly assertable.

4. GREEN — the least code that passes

Now write just enough to satisfy the test. bash's $RANDOM spans [0, 32767] — that's 32768 distinct values. Go's math/rand/v2 gives us exactly the right tool: rand.IntN(n) returns an int in the half-open interval [0, n). So rand.IntN(32768) yields 0 through 32767. Create rndm.go: 

// rndm.go
package main

import "math/rand/v2"

// Number returns a pseudo-random integer in [0, 32767],
// matching the range of bash's $RANDOM.
func Number() int {
	return rand.IntN(32768) // IntN(n) yields [0, n)
}

Run the tests again:

go test ./...
ok  	github.com/<you>/rndm	0.002s

Green. Want to watch it actually run? Use -v:

go test -v ./...
=== RUN   TestNumberIsInBashRandomRange
--- PASS: TestNumberIsInBashRandomRange (0.00s)
PASS
ok  	github.com/<you>/rndm	0.003s
Why math/rand/v2, and no seeding In the old math/rand you had to call rand.Seed(...) or every run produced the same sequence. v2 (Go 1.22+) fixed that: its top-level functions are automatically seeded with a random value and are safe for concurrent use — so each run differs with zero ceremony. One caveat worth knowing: IntN(n) panics if n ≤ 0. Not a concern with our constant 32768, but remember it for when the bound becomes a user-supplied flag.

5. Make it a program — main() and stdout

The test drove Number(), the logic. But a CLI has to actually print something. Add a main that writes the number to standard output. This is where clig.dev earns its place: "Send output to stdout", and "Return zero exit code on success." A Go program that returns from main normally exits 0 automatically. 

// rndm.go
package main

import (
	"fmt"
	"math/rand/v2"
	"os"
)

// Number returns a pseudo-random integer in [0, 32767],
// matching the range of bash's $RANDOM.
func Number() int {
	return rand.IntN(32768)
}

func main() {
	fmt.Fprintln(os.Stdout, Number())
}
Why Fprintln(os.Stdout, …) and not just Println? fmt.Println is a shortcut that always writes to os.Stdout. Writing os.Stdout explicitly names the stream we mean — output goes to stdout, and if we ever needed diagnostics they'd go to os.Stderr. It also foreshadows Lesson 2: the moment that destination becomes a parameter (io.Writer), we can capture it in a test. Keeping main() this thin — parse nothing, compute in Number(), print — is what keeps the logic testable.

Build the binary and run it a few times — a different number each run:

go build -o rndm .
./rndm
./rndm
./rndm
26068
31348
8272

Confirm it behaves as a good CLI citizen — clean on a pipe, and exit code 0:

./rndm | cat        # just the number, nothing else on stdout
./rndm > /dev/null; echo "exit=$?"
6765
exit=0

Per clig.dev, where should rndm's random number go, and what exit code signals success?

Result on stdout, success is 0. stdout carries the actual output so it can become another program's input; stderr is for messages and errors; a zero exit code tells a calling script the run succeeded. That trio is the baseline contract of a well-behaved CLI.

6. REFACTOR — and keep the tests honest

With both tests green, this is the safe moment to improve the code. There's little to tidy in something this small — but the refactor step is also where you run the housekeeping tools and trust the suite to catch any regression: 

go vet ./...     # static checks — should print nothing
gofmt -l .       # lists unformatted files — empty means all good
go test ./...    # still green?

That's one complete turn of the cycle: red (a failing test named the behaviour) → green (least code to satisfy it) → refactor (tidy with a safety net). Every feature you add to rndm from here follows the same loop.

Main takeaways
  • Test-first: write a failing test that defines "done", then write code to pass it. A compile error is a legitimate red.
  • Testing randomness: you can't assert an exact value — assert a property (here, the result is always in [0, 32767]) across many draws.
  • rand.IntN(n) from math/rand/v2 returns [0, n); it panics if n ≤ 0. v2 is auto-seeded — no rand.Seed needed.
  • Thin main(): keep logic in a testable function (Number()); let main just wire input → compute → output.
  • clig.dev contract: result to stdout, messages to stderr, exit 0 on success — so the tool pipes and scripts cleanly.
  • The loop that carries the whole course: red → green → refactor.
