Complete vs Partial Rules

A one-page card. The two rule shapes, side by side. Compressed essence of Lessons 1–2.

The one distinction A complete rule produces one value (or nothing). A partial rule produces a collection, built up across definitions. Which "empty" you get when nothing matches is the giveaway: undefined vs [].

Side by side

Complete rulePartial set rule
Question "Is it allowed?" — one answer "What's wrong?" — a list
Syntax allow if <body> deny contains msg if <body>
Produces a single value a set (unions all definitions)
No body holds undefined — key absent [] (defined, empty)
Needs default? usually — to answer when undefined never — empty is the answer
Two definitions must agree, or it's a conflict always merge (union)
Used for a verdict: allow, deny-as-bool findings: conftest deny/warn

Canonical shapes (OPA 1.0)

# COMPLETE — one value, default fallback
package example
default allow := false
allow if input.age >= 18


# PARTIAL SET — many findings, no default needed
package signup
violations contains "age must be at least 18" if input.age < 18
violations contains "username is required"   if input.username == ""

# fold findings back into a verdict
deny if count(violations) > 0

The conftest connection

Same shape, different name Conftest scans package main for partial set rules named deny, warn, or violation. It collects each set, prints the messages, and fails when deny is non-empty. Everything on this card stays true there — conftest is just OPA collecting a set for you.
