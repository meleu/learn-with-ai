Rego Mental Model & Toolbox

A one-page card to revisit. Compressed essence of Lesson 1.

The one-sentence model

A .rego file is not a script that runs top-to-bottom — it is a set of declarative rules that build a JSON document. You hand OPA some JSON input; it evaluates your rules; it hands a JSON document back. A decision is just a value sitting in that document. OPA decides; something else enforces.

Glossary

Policy decoupling
Moving policy decisions out of application code into a dedicated engine (OPA). The reason OPA exists.
Decision
Not a special thing — just a value the rules put in the document (e.g. allow: true) that you read back. OPA never acts on it.
Enforcement
Acting on the decision — done by the caller (your app, your pipeline).
input
One root of the tree: the JSON for this question, supplied per query with opa eval -i.
data
The other root: pre-loaded context you supply (config, lookup tables) plus the documents your rules generate. Your policy's output lives under data.
Rule
A definition that generates content in the document — it contributes a value to the tree when its body is satisfied. The unit everything is built from.
Rule body
Conditions joined by AND. OPA looks for variable bindings that make them all true.
Undefined
What a rule contributes when its body can't be satisfied: nothing — the key is simply absent from the document. Not false or null (those are real values that are in the document).
Complete rule
Produces a single value (e.g. allow). Conflicting values → error.
Partial rule
Builds a set or object incrementally; multiple definitions union together. (Later lesson.)
Package
The folder in the tree your rules write into. The rule allow in package example lives at data.example.allow.
default
Fallback that guarantees the key is always present — turns an absent (undefined) key into a real value like false when no body fires.

Canonical policy shape (OPA 1.0)

package example

# fallback so the rule always answers
default allow := false

# allow is true when the body can be satisfied
allow if input.age >= 18

if = the rule has a body · constants (default …) need no if · no default ⇒ a failing body leaves the key absent (undefined), not false.

The three outcomes of a complete rule

true
Body satisfied → the rule adds its value to the document (allow: true).
false
A real, present value — only if a default (or another body) supplies it.
undefined
Body fails and no default → the rule adds nothing; the key is absent from the document (whole tree comes back {}). Querying that exact path prints undefined.

CLI cheat sheet (opa)

# evaluate a decision
opa eval -d policy.rego -i input.json 'data.example.allow' --format pretty
opa eval -d policy.rego -i input.json 'data.example'   # whole doc tree

# interactive REPL (great for exploring)
opa run policy.rego                        # then type expressions

# format & test
opa fmt -w policy.rego                     # rewrite to 1.0 style
opa test .                                 # run *_test.rego unit tests
opa version
Debugging tip Got undefined when you expected a value? Ask for the parent path (data.example) or a sub-expression to see how far evaluation got. Most surprises are a wrong path into input, not wrong logic.
