Rego Mental Model & Toolbox

A one-page card to revisit. Compressed essence of Lesson 1.

The one-sentence model

A .rego file is a set of declarative definitions that, given an input document, computes a decision. It decides; something else enforces.
input
JSON in
→
rules
your policy
→
decision
value out

Glossary

Policy decoupling
Moving policy decisions out of application code into a dedicated engine (OPA). The reason OPA exists.
Decision
The value OPA computes (e.g. allow = true). OPA never acts on it.
Enforcement
Acting on the decision — done by the caller (your app, your pipeline).
input
The document being evaluated, supplied per query with opa eval -i.
data
Persistent, pre-loaded context (config, lookup tables) available to every rule.
Rule
A definition that generates the content of a document. The unit everything is built from.
Rule body
Conditions joined by AND. OPA looks for variable bindings that make them all true.
Undefined
The result when a rule body can't be satisfied. Not false — it means "no value / doesn't apply".
Complete rule
Produces a single value (e.g. allow). Conflicting values → error.
Partial rule
Builds a set or object incrementally; multiple definitions union together. (Later lesson.)
Package
Namespace for a module. The rule allow in package example lives at data.example.allow.
default
Fallback value for a complete rule when all its bodies are undefined.

Canonical policy shape (OPA 1.0)

package example

# fallback so the rule always answers
default allow := false

# allow is true when the body can be satisfied
allow if input.age >= 18

if = the rule has a body · constants (default …) need no if · no default ⇒ a failing body yields undefined, not false.

The three outcomes of a complete rule

true
Body satisfied → rule takes its value.
false
Only if a default (or another body) supplies it.
undefined
Body fails and no default → no value at all. opa eval --format pretty prints undefined.

CLI cheat sheet (opa)

# evaluate a decision
opa eval -d policy.rego -i input.json 'data.example.allow' --format pretty
opa eval -d policy.rego -i input.json 'data.example'   # whole doc tree

# interactive REPL (great for exploring)
opa run policy.rego                        # then type expressions

# format & test
opa fmt -w policy.rego                     # rewrite to 1.0 style
opa test .                                 # run *_test.rego unit tests
opa version
Debugging tip Got undefined when you expected a value? Ask for the parent path (data.example) or a sub-expression to see how far evaluation got. Most surprises are a wrong path into input, not wrong logic.
Conftest — coming later Conftest wraps OPA: it parses config files into input and looks for rules named deny/warn/violation in package main. It's the pipeline tool, but it's just OPA underneath — this card stays true when you get there.
