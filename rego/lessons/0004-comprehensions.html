Comprehensions

Iteration that hands you back data. Collect the values themselves — a set of names, a lookup map — not just a yes/no.

Your win for this lesson By the end you can (1) read and write the three comprehension forms — set {v | …}, array [v | …], and object {k: v | …} — (2) know when order and duplicates matter (array vs set), and (3) use a comprehension inside a rule to compute with the values you collected.

1. From verdicts to values

In Lesson 3 iteration produced two things: some + contains gave a set of findings (strings), and every gave a single verdict (a boolean). Both throw away the values along the way — you learn that something is wrong, or how many, but not the values as data you can keep working with.

A comprehension is the third thing iteration can do: it builds a new collection from the elements you iterate. The set of offending container names. A map of name → image. The list of every exposed port. You get back a value you can count, compare, feed into sprintf, or test in another rule. It's the same some you already know — pointed at collecting instead of deciding.

some c in xs
iterate the elements
→
{ value | body }
keep value where body holds
→
a collection
set · array · object

2. The set comprehension — { value | body }

Read { value | body } aloud as "the set of value, for every binding where body holds." The bar | separates what to collect (on the left) from which bindings qualify (the body on the right — the very same kind of some + conditions body from Lesson 3). This one collects the name of every privileged container: 

package manifest

# the SET of c.name, for every container that runs privileged
privileged_names := {c.name |
	some c in input.containers
	c.securityContext.privileged == true
}

This is a complete rule (Lesson 1) whose value happens to be a set — not a contains partial rule. Feed it a manifest where two of three containers are privileged: 

# input.json
{
  "containers": [
    { "name": "api",   "image": "corp/api:1.4.2", "securityContext": { "privileged": true } },
    { "name": "cache", "image": "redis:latest" },
    { "name": "web",   "image": "nginx:latest",  "securityContext": { "privileged": true } }
  ]
}
opa eval -d manifest.rego -i input.json 'data.manifest.privileged_names' --format pretty

A set prints as a JSON array (sets have no duplicates and no order):

[
  "api",
  "web"
]
left of |
The value to collect for each qualifying binding — here c.name.
right of |
The body: iterate with some, then any conditions (AND), exactly as in a rule.
{ … }
Braces around a single value ⇒ a set comprehension (dedupes, unordered).
Gotcha — the linter will flag this Regal, Rego's linter, flags a complete rule whose whole value is a comprehension with prefer-set-or-object-rule. It would rather you wrote the equivalent partial rule — the contains shape from Lesson 2: 
privileged_names contains c.name if {
	some c in input.containers
	c.securityContext.privileged == true
}
Both compute the same set. We assign the comprehension to a rule here on purpose, to keep the { value | body } syntax front and centre — but in real policies, prefer the partial rule.

3. Array vs set — when order and duplicates matter

Swap the braces for square brackets and you get an array comprehension: same body, but it keeps every value in order, duplicates included. 

# every image, in list order, duplicates kept
all_images := [c.image | some c in input.containers]

# the distinct images, deduped, unordered
distinct_images := {c.image | some c in input.containers}

That difference is a tool, not a nuisance. Comparing the two counts is the idiomatic way to ask "are these values all distinct?" — because the array keeps duplicates and the set collapses them: 

default names_unique := false
names_unique if {
	all_names := [c.name | some c in input.containers]        # keeps dups
	distinct_names := {c.name | some c in input.containers}   # drops dups
	count(all_names) == count(distinct_names)                # equal ⇒ no duplicate names
}

Three containers named web, web, db give count(all_names) == 3 but count(distinct_names) == 2, so names_unique is false. Rename one and the counts match. Pick the array when order or repetition carries meaning; pick the set when you want a clean membership list.

A manifest has containers named web, web, db. What is count([c.name | some c in input.containers])?

Three. An array comprehension keeps one entry per binding, duplicates and all — so the count equals the number of elements. Only the set form ({ … }) would collapse the two webs and give two.

4. The object comprehension — { key: value | body }

Put a colon in the head — key: value — and the braces build an object (a map) instead of a set. Perfect for turning a list into a lookup keyed by something meaningful: 

# a lookup: container name → its image
image_by_name := {c.name: c.image | some c in input.containers}
opa eval -d manifest.rego -i input.json 'data.manifest.image_by_name' --format pretty

An object prints as JSON (keys sorted):

{
  "api": "corp/api:1.4.2",
  "cache": "redis:latest",
  "web": "nginx:latest"
}

Now image_by_name["web"] reads the image directly, no re-iteration. One caution: the keys must be unique. If two bindings produce the same key with different values, OPA raises a conflict error rather than silently picking one — which is usually a signal your key isn't as unique as you assumed.

Gotcha — the linter again Just like the set example in §2, assigning an object comprehension to a complete rule trips Regal's prefer-set-or-object-rule. The idiomatic form is a partial object rule: 
image_by_name[c.name] := c.image if {
	some c in input.containers
}
Same map, idiomatic shape. We keep the comprehension here to show the { key: value | body } syntax.

5. Comprehensions earn their keep inside a rule

The real payoff is using a comprehension as a step in a rule body: collect the values, then decide or report with them. Here, instead of one finding per privileged container (Lesson 3), we collect the names first and emit a single, tidy finding that lists them all: 

violations contains msg if {
	names := {c.name |                          # collect, as a step in the body
		some c in input.containers
		c.securityContext.privileged == true
	}
	count(names) > 0                           # only report if there are any
	msg := sprintf("privileged containers: %s", [concat(", ", names)])
}
# → "privileged containers: api, web"

concat(sep, coll) joins a set or array of strings with a separator (sorted, for a set). Note the count(names) > 0 guard: a comprehension never goes undefined — with no matching bindings it simply yields an empty collection. So names would be the empty set, concat would give "", and without the guard you'd emit a hollow "privileged containers: " finding. Collecting always succeeds; deciding whether an empty result is worth reporting is still your call.

No container is privileged. What does the comprehension {c.name | some c in input.containers; c.securityContext.privileged == true} evaluate to?

The empty set. A comprehension always produces a collection; zero matching bindings just means zero elements. That's why you guard with count(…) > 0 when an empty result shouldn't count as a finding — the comprehension won't skip the rule for you the way an undefined value would.

Main takeaways
  • A comprehension turns iteration into a collection — the values themselves, not just a finding or a boolean.
  • Read { value | body } as "the collection of value for every binding where body holds"; the body is a normal some + conditions body.
  • Set {v | …} dedupes and is unordered; array [v | …] keeps order and duplicates — compare their counts to test for duplicates.
  • Object {k: v | …} builds a lookup map; keys must be unique or OPA raises a conflict.
  • A comprehension is a natural step inside a rule body — collect, then decide or sprintf with the result.
  • A comprehension never goes undefined: no matches ⇒ an empty collection, so guard with count(…) > 0 when empty shouldn't report.
