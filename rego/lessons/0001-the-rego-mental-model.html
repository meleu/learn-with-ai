The Rego Mental Model

Why Rego exists, what a .rego file really is, and the tools you'll practice with.

Your win for this lesson By the end you can (1) explain to a coworker why OPA/Rego exists, (2) describe a .rego file as a function from data to a decision, and (3) run your first policy locally with opa eval.

1. Why Rego was created

Every system already makes policy decisions: is this user an adult? can they delete that resource? does this image come from a trusted registry? The old way is to bury those decisions inside each application's code — scattered, inconsistent, and impossible to change without a redeploy.

OPA's founding idea is policy decoupling: pull the decision out of the application and into a dedicated engine. The official philosophy puts it plainly — policies become something that can be "specified declaratively, updated at any time without recompiling or redeploying, and enforced automatically."

The distinction that unlocks everything Decision vs. enforcement. OPA decides ("this request is not allowed"). The calling system enforces ("so I'll reject it"). Rego only ever computes the decision — it never blocks, deletes, or deploys anything itself.

For you, owner of pipelines building hundreds of microservices, the payoff is concrete: one versioned, testable set of rules — policy as code — instead of the same checks hand-copied into every repo. Without a tool like this you'd have to "implement policy management for your software from scratch": invent a language, an evaluator, tests, the works.

A Rego policy decides a request is not allowed. What happens to the request?

Decision, not enforcement. Rego computes a verdict from the data it's given. Acting on that verdict — rejecting the request, failing the pipeline — is the caller's job. This separation is the whole point of OPA.

2. What a .rego file actually is

Here is the mental model to hold onto, and the one thing most newcomers get wrong. A .rego file is not a script that runs top-to-bottom doing things. It is a set of declarative definitions. Given some input data, it computes a value. Think of it as a pure function:

input
the JSON you're asking about, e.g. {"age": 21}
→
policy (rules)
your .rego definitions
→
decision
a value: boolean, message, object…

OPA sees the world as JSON documents. Two of them matter now:

input
The data for this question, handed in by the caller at query time.
data
Persistent, pre-loaded context available everywhere (e.g. a legal minimum age, a list of approved registries). Optional; you may not need it at first.

Everything in the file is built from rules. The docs define them precisely: "rules define the content of virtual documents… When OPA evaluates a rule, OPA generates the content of the document that is defined by the rule." Here is a complete policy — an age check. Read it as a definition, not a procedure: 

package example

# allow is false unless a rule below proves otherwise
default allow := false

# allow becomes true when the applicant is an adult
allow if input.age >= 18

That's the whole policy. Two pieces beyond the package:

allow if …
A rule. Its body (input.age >= 18) is the condition. OPA "searches for variable bindings that make all of the expressions true." If it can, allow gets the value true.
default allow
The fallback. When the rule body can't be satisfied, the rule produces no value — it's undefined. default supplies false so the policy always answers.

Notice what's absent: no loop, no else ladder, no return. You describe what makes allow true; OPA does the searching. And note the crucial subtlety: without that default line, an applicant aged 16 doesn't make allow false — it makes it undefined. Undefined is Rego's quiet "this doesn't apply / no answer", and it is not the same as false.

With the default line removed, what is allow for {"age": 16}?

Undefined, not false. input.age >= 18 can't be made true, so the rule generates nothing. allow is undefined. Adding default allow := false is what turns "no answer" into a usable false.

3. Run it with opa

The document a rule generates lives at data.<package>.<rule> — here, data.example.allow. That path is how you ask OPA for the decision. Create two files: 

# policy.rego
package example

default allow := false

allow if input.age >= 18
# input.json
{ "age": 21 }

Then evaluate the decision:

opa eval -d policy.rego -i input.json 'data.example.allow' --format pretty

Output:

true

Change age to 16 and re-run: you get false (thanks to default). Delete the default line and run the 16 case again: you get undefined — exactly the quiz above, now on your own machine. That loop — edit input, re-run, watch the decision change — is how you'll learn Rego.

Read the whole document Ask for the package instead of one rule — opa eval -d policy.rego -i input.json 'data.example' — and OPA prints every document the file generates: { "allow": true }. A .rego file is the document tree it defines.

4. The tools to practice with

Rego Playground
Browser REPL. Paste policy + input, see the decision instantly. Zero install — the fastest way to experiment and to share a snippet with a coworker.
opa
The engine, and your main tool for this course. opa eval (evaluate any expression), opa fmt (auto-format to current syntax), opa test (unit-test policies).
Where conftest fits (later) Conftest is a thin wrapper that runs OPA for you: it parses config files (YAML, Dockerfile, HCL…) into input and applies a naming convention. It's your eventual pipeline tool — but it's just OPA underneath. Learn the engine first; we'll add conftest once the Rego fundamentals are solid.

Install opa with Homebrew

We use Homebrew for tool installs — including on Linux — so everyone on your team runs the same one-liner and gets updates the same way: 

brew install opa
opa version
Syntax note — OPA 1.0 Since OPA 1.0, if before a rule body is the default, required syntax (constants like default allow := false need no if). Older tutorials omit it — run opa fmt and it'll flag them. We'll always use the modern form.

Which command asks OPA for the decision computed by the allow rule?

opa eval evaluates an expression. -d loads the policy, -i supplies input, and the query data.example.allow is the path to the document your rule generates.

Main takeaways
  • Policy decoupling is the reason Rego exists: pull decisions out of application code into a versioned, testable engine.
  • Decision vs. enforcement. Rego only computes a verdict; the caller acts on it. Rego never blocks, deletes, or deploys.
  • A .rego file isn't a script — it's declarative definitions that compute a value from input, like a pure function.
  • input is the data for this question; data is persistent, pre-loaded context.
  • A rule (allow if …) yields its value when OPA can satisfy the body; default supplies a fallback so the policy always answers.
  • Undefined ≠ false. Without a default, an unsatisfied body produces no value, not false.
  • Ask OPA for a decision at data.<package>.<rule> with opa eval -d policy.rego -i input.json 'data.example.allow'.
  • The core learning loop: edit input → re-run → watch the decision change.
