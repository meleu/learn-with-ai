The Rego Mental Model

Why Rego exists, and the one idea that unlocks it: a .rego file is a program that builds a JSON document.

Your win for this lesson By the end you can (1) explain to a coworker why OPA/Rego exists, (2) describe a .rego file as rules that build a JSON document from input data, and (3) run your first policy locally with opa eval and read the document it produced.

1. Why Rego was created

Every system already makes policy decisions: is this user an adult? can they delete that resource? does this image come from a trusted registry? The old way is to bury those decisions inside each application's code — scattered, inconsistent, and impossible to change without a redeploy.

OPA's founding idea is policy decoupling: pull the decision out of the application and into a dedicated engine. The official philosophy puts it plainly — policies become something that can be "specified declaratively, updated at any time without recompiling or redeploying, and enforced automatically."

The distinction that unlocks everything Decision vs. enforcement. OPA decides ("this request is not allowed"). The calling system enforces ("so I'll reject it"). Rego only ever computes the decision — it never blocks, deletes, or deploys anything itself.

For you, owner of pipelines building hundreds of microservices, the payoff is concrete: one versioned, testable set of rules — policy as code — instead of the same checks hand-copied into every repo. Without a tool like this you'd have to "implement policy management for your software from scratch": invent a language, an evaluator, tests, the works.

A Rego policy decides a request is not allowed. What happens to the request?

Decision, not enforcement. Rego computes a verdict from the data it's given. Acting on that verdict — rejecting the request, failing the pipeline — is the caller's job. This separation is the whole point of OPA.

2. What a .rego file actually is

Here is the mental model to hold onto — and the one thing most newcomers get wrong. A .rego file is not a script that runs top-to-bottom doing things. It is a set of declarative rules that build a JSON document. You hand OPA some JSON in; it evaluates your rules; it hands a JSON document back.

input
JSON you're asking about, e.g. {"age": 21}
→
rules
your .rego definitions
→
a JSON document
what the rules built, e.g. {"allow": true}

This is the whole game, and the docs say it in exactly these terms: rules "define the content of virtual documents… When OPA evaluates a rule, OPA generates the content of the document that is defined by the rule." A "decision" is not a special kind of thing — it is just a value sitting somewhere in the JSON document your rules built.

Everything OPA knows lives in one big JSON tree. Two roots matter now:

input
The data for this question, handed in by the caller at query time.
data
Everything else: pre-loaded context you supply (a legal minimum age, a list of approved registries) plus the documents your rules generate. Your policy's output lives under data.

Here is a policy — an age check. Don't read it as a procedure; read it as a description of what the output document should contain: 

package example

# put allow: true in the document when the applicant is an adult
allow if input.age >= 18

With input of {"age": 21}, the document this rule builds looks like this — your rule generated the example.allow node: 

data
└─ example
   └─ allow: true      # ← the value your rule generated
allow if …
A rule. Its body (input.age >= 18) is the condition under which it contributes a value. OPA "searches for variable bindings that make all of the expressions true." If it succeeds, the document gets allow: true.

Notice what's absent: no loop, no else ladder, no return. You describe what belongs in the document; OPA does the searching to build it. The package name (example) is just the folder in the tree your rules write into.

For input {"age": 21}, what does the rule add to the output document?

Rules build documents. The rule doesn't "return" or "print" — it contributes a value to the tree. Here it generates allow: true under data.example. You then read that node to get the decision.

3. Run it — and read the document

Every value a rule builds is addressable by its path in the tree: data.<package>.<rule> — here, data.example.allow. That path is how you ask OPA for a piece of the document. Create two files: 

# policy.rego
package example

allow if input.age >= 18
# input.json
{ "age": 21 }

First, ask for the whole document the policy built:

opa eval -d policy.rego -i input.json 'data.example' --format pretty
{
  "allow": true
}

That is the mental model made literal — the file is the JSON document it defines. Now ask for just one node, the decision: 

opa eval -d policy.rego -i input.json 'data.example.allow' --format pretty
true

That loop — edit input, re-run, read the document — is how you'll learn Rego. Now try one small change before moving on: set age to 16 and re-run. The result is the one thing nearly every newcomer trips on.

4. The one subtlety: undefined is a missing key

This is the idea most newcomers stumble on, and the document model explains it cleanly. With age set to 16, ask for the whole document: 

opa eval -d policy.rego -i input.json 'data.example' --format pretty
{}

Not {"allow": false}. Not an error. An empty document. The body input.age >= 18 couldn't be satisfied, so the rule contributed nothing — the allow key simply isn't there. Rego calls this undefined, and structurally it means exactly "this key is absent from the document." It is not the same as false or null, which are real values that are in the document.

For {"age": 16}, with only the allow if rule, the allow key is…

Undefined is absence. An unsatisfied body contributes no value, so the key never appears. That's why the document comes back {}. default turns "no key" into a present, usable false.

Guaranteeing the key: default

An empty document is rarely what a caller wants — asking "am I allowed?" and getting nothing back is awkward to handle. The fix is a single line, default, which supplies a value for when no rule fires: 

package example

# fall back to false unless a rule below proves otherwise
default allow := false

allow if input.age >= 18

Re-run with age still 16 — now the document holds a real, present false:

opa eval -d policy.rego -i input.json 'data.example' --format pretty
{
  "allow": false
}
default allow
The fallback value. When the rule body can't be satisfied it contributes nothing, so default fills the gap with false — the allow key is now always present and callers never see undefined. Being a constant, it needs no if.
Two honest asterisks on "it's all JSON" (1) The JSON data model, not literal JSON text. Your input needn't be a .json file — later, conftest parses YAML, Dockerfiles, and HCL into the same tree. And Rego adds one type JSON lacks: sets (you'll build these in Lesson 2), which OPA serializes to arrays on output. (2) undefined is the one thing that is not a value — it's the absence of a key, as you just saw. Everything else OPA produces is ordinary JSON.

5. The tools to practice with

Rego Playground
Browser REPL. Paste policy + input, see the document instantly. Zero install — the fastest way to experiment and to share a snippet with a coworker.
opa
The engine, and your main tool for this course. opa eval (evaluate any path in the document), opa fmt (auto-format to current syntax), opa test (unit-test policies).
Where conftest fits (later) Conftest is a thin wrapper that runs OPA for you: it parses config files (YAML, Dockerfile, HCL…) into input and applies a naming convention. It's your eventual pipeline tool — but it's just OPA underneath. Learn the engine first; we'll add conftest once the Rego fundamentals are solid.

Install opa with Homebrew

We use Homebrew for tool installs — including on Linux — so everyone on your team runs the same one-liner and gets updates the same way: 

brew install opa
opa version
Syntax note — OPA 1.0 Since OPA 1.0, if before a rule body is the default, required syntax (constants like default allow := false need no if). Older tutorials omit it — run opa fmt and it'll flag them. We'll always use the modern form.

Which command prints the whole JSON document that policy.rego builds?

opa eval evaluates a path. -d loads the policy, -i supplies input, and the query data.example is the node in the tree your package writes into — so OPA prints the document it built there.

Main takeaways
  • Policy decoupling is the reason Rego exists: pull decisions out of application code into a versioned, testable engine.
  • Decision vs. enforcement. Rego only computes a verdict; the caller acts on it. Rego never blocks, deletes, or deploys.
  • A .rego file isn't a script — it's rules that build a JSON document from input.
  • Everything lives in one JSON tree: input (this question's data) and data (loaded context plus the documents your rules generate).
  • A rule (allow if …) contributes a value to the tree when OPA can satisfy its body; default supplies a fallback so the key is always present.
  • Undefined = a missing key. An unsatisfied body adds nothing, so the key is absent — not false, not null.
  • Read any part of the document at data.<package>.<rule> with opa eval -d policy.rego -i input.json 'data.example'.
  • The core learning loop: edit input → re-run → read the document.
