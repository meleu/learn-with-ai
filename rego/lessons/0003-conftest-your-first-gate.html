Conftest: Your First Real Gate

The exact partial rule from Lesson 2, pointed at a real Deployment — and an exit code that fails the build.

Your win for this lesson By the end you can (1) write a working conftest policy using nothing but Lesson 2's partial rule, (2) run it on a real YAML file and read the FAIL lines and the exit code, and (3) spot a check that can silently never fire.

1. A decision isn't yet enforcement

Lesson 2 ended one step short. You collected findings into a set, folded them into a verdict with deny if count(violations) > 0 — and then that verdict just sat there, printed in your terminal by opa eval. Nothing was gated. Nothing failed.

Conftest is the missing half. It's the tool from your mission — the thing that actually runs in your pipelines. And it changes only two things about what you already do. No new Rego at all.

Change 1
input is your config file, parsed for you — no more hand-written input.json, no -i flag. YAML, JSON, Dockerfile, HCL, TOML and more, picked by file type. (conftest.dev)
Change 2
Rule names are a convention. Conftest collects the rules called deny, warn, and violation from package main. That's the whole API. The rule bodies stay the Rego you already write.

2. deny is your Lesson 2 rule, renamed

Here is a real policy for a Kubernetes Deployment — three checks a platform team genuinely enforces. Read it before reading the explanation, and notice that you already know every line: 

# policy/deployment.rego
package main

deny contains "Deployment must not run in the default namespace" if input.metadata.namespace == "default"

deny contains "Deployment must have at least 2 replicas" if input.spec.replicas < 2

deny contains "Deployment is missing the owner label" if not input.metadata.labels.owner

That is the Lesson 2 signup validator with the strings swapped. Same three shapes, same union, same "no findings means []". Line for line:

input.age < 18input.spec.replicas < 2comparison

input.username == ""input.metadata.namespace == "default"equality

Only two things are conftest's ask: the package must be main, and the rule must be called deny. Everything else is the partial set rule you already write.

Why input.metadata.namespace and not input.namespace? Conftest hands you the whole parsed file as input, so your paths follow the file's real shape. A Deployment nests its namespace under metadata, so that's the path. Same mental model as Lesson 1: input is just a document, and you reach into it exactly as it's written.

3. Run it on a real file

Conftest reads policies from a policy/ directory by default. Put the rules there and drop a manifest beside it: 

.
├── deployment.yaml     # the config to check — becomes `input`
└── policy/
    └── deployment.rego # package main
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: billing
  namespace: default
  labels:
    team: payments
spec:
  replicas: 1
  template:
    spec:
      containers:
        - name: api
          image: corp/api:1.4.2
conftest test deployment.yaml

All three checks trip — one FAIL line per element of your deny set:

FAIL - deployment.yaml - main - Deployment is missing the owner label
FAIL - deployment.yaml - main - Deployment must have at least 2 replicas
FAIL - deployment.yaml - main - Deployment must not run in the default namespace

3 tests, 0 passed, 0 warnings, 3 failures, 0 exceptions

And now the piece that makes it a gate:

echo $?     # → 1   (any deny finding ⇒ non-zero exit)

Fix the manifest — namespace: payments, replicas: 3, an owner label — and the same command reports 3 tests, 3 passed, 0 failures and exits 0.

This is the enforcement point Lesson 1 drew the line between deciding (OPA's job) and enforcing (someone else's). That "someone else" is now concrete: conftest exits non-zero when any deny fires, and your CI step fails the build. The Rego decides; the exit code enforces. Everything else in this course is about making that decision correct.

conftest test prints two FAIL lines. What does the command exit with?

One. The exit code is a gate, not a tally: it is 1 if the deny set is non-empty and 0 if it's empty, no matter how many findings there are. That single bit is what your CI step reads.

4. The check that can never fire

Now a trap, and it's the most valuable thing in this lesson. Take a Deployment that is otherwise clean but omits spec.replicas entirely — very common, since Kubernetes defaults it to 1: 

spec:
  # no replicas key at all
  template: ...
conftest test noreplicas.yaml

3 tests, 3 passed, 0 warnings, 0 failures, 0 exceptions     # exit 0

It passes. A Deployment that will run a single replica sails straight through the gate you wrote to prevent exactly that. Nothing errored, nothing warned — the check simply never fired, because input.spec.replicas is undefined, so undefined < 2 is undefined, so the body can't be satisfied (Lesson 1).

The fix is Lesson 2's missing-versus-empty username pair, transplanted whole: a second definition guarding the other failure mode. 

# the value is wrong…
deny contains "Deployment must have at least 2 replicas" if input.spec.replicas < 2

# …and the key is missing — two failure modes, two definitions, one set
deny contains "Deployment must set spec.replicas explicitly" if not input.spec.replicas
FAIL - noreplicas.yaml - main - Deployment must set spec.replicas explicitly

4 tests, 3 passed, 0 warnings, 1 failure, 0 exceptions      # exit 1
The habit to build now Undefined skips the rule is a safety feature when you're reaching into optional nested structure — it stops your policy crashing on manifests that don't have the field. But for a field you require, that same behaviour is a silent hole. Every time you write a check on a value, ask the second question: what if the key isn't there at all? A gate that can't fire is worse than no gate, because everyone believes it's working.

A Deployment omits spec.replicas. What does deny contains … if input.spec.replicas < 2 do?

Nothing fires. Reaching a missing key yields undefined, and any comparison against undefined is itself undefined, so the body can't be satisfied and the rule contributes no finding. No error, no warning — the manifest just passes. Add a not input.spec.replicas definition to catch the absent case.

5. deny vs warn vs violation

Conftest recognises three rule names, differing only in how hard they push and, for one, the shape of the message (conftest.dev):

RuleMessageEffect on the build
denya stringfails — exit code 1
warna stringprints a warning, build still passes — exit 0
violationa structured objectsame as deny, but returns rich data, not just text

Reach for warn when something is worth flagging but shouldn't block a merge. Demote the owner-label check and the finding still surfaces — but the gate opens: 

warn contains "Deployment has no owner label" if not input.metadata.labels.owner
WARN - deployment.yaml - main - Deployment has no owner label

4 tests, 3 passed, 1 warning, 0 failures, 0 exceptions      # exit 0

A run reports 0 failures but 2 warnings. What exit code does conftest return?

Zero. Only deny and violation findings make conftest exit non-zero. warn exists precisely to surface an issue without failing the build — the step passes and CI moves on. Use it for advisories, not hard gates.

6. Name your rules so a coworker can read them

Four bare deny blocks work, but the mission is also to teach coworkers. Conftest lets any deny/warn/violation rule carry an underscore suffix and still collects it (conftest.dev). The suffix is documentation; conftest ignores it when matching. Here's the finished policy: 

package main

deny_default_namespace contains "Deployment must not run in the default namespace" if input.metadata.namespace == "default"

deny_replicas_unset contains "Deployment must set spec.replicas explicitly" if not input.spec.replicas

deny_replicas_too_low contains "Deployment must have at least 2 replicas" if input.spec.replicas < 2

warn_no_owner contains "Deployment has no owner label" if not input.metadata.labels.owner
WARN - deployment.yaml - main - Deployment has no owner label
FAIL - deployment.yaml - main - Deployment must not run in the default namespace
FAIL - deployment.yaml - main - Deployment must have at least 2 replicas

4 tests, 1 passed, 1 warning, 2 failures, 0 exceptions      # exit 1

Prefer one clearly-named rule per concern over one deny with a tangled body. A coworker skimming the file learns what you enforce without reading a single condition.

One syntax detail real manifests force on you input.metadata.labels.owner works because owner is a plain key. Real Kubernetes labels usually aren't — app.kubernetes.io/name contains dots, which Rego would read as path steps. Reach for such a key with brackets: input.metadata.labels["app.kubernetes.io/name"]. Same lookup, quoted key.
Next: manifests that are lists Every check here read a fixed field, so one definition per problem was enough. But a Deployment's real risk lives in spec.template.spec.containers — a list, of unknown length, where you need a finding naming which container is at fault. That's Lesson 4: one rule that fans out over a list.

Main takeaways
  • Conftest is OPA with two conveniences: it parses your config file into input, and it collects rules by name.
  • A deny contains … if rule is the Lesson 2 partial set rule — just named deny inside package main.
  • A useful policy needs no iteration and no new syntax: comparison, presence, and equality on fixed fields already gate real manifests.
  • input is the whole parsed file, so paths follow its real shape (input.metadata.namespace, not input.namespace).
  • Each finding is one FAIL line; the exit code is the enforcement point — 1 if deny is non-empty, 0 if empty.
  • A check on a missing key never fires — a silently-passing gate. Pair every value check with a not input.… presence check.
  • deny fails the build; warn advises and passes; violation is deny with structured data.
  • Suffix rules (deny_replicas_too_low) to make each check self-documenting for coworkers.
