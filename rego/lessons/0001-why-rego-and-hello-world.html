Why Rego Exists — and Your First Policy

Lesson 0001 · ~15 minutes · Mission: conftest policies at work

1 · Why was Rego created?

Every service you run makes policy decisions: may this deployment run as root? Does this pipeline pin its image versions? Is this pom.xml using an approved repository? Traditionally those rules live hard-coded inside each tool or service — scattered, inconsistent, and impossible to update without redeploying software.

Open Policy Agent (OPA) was created to fix exactly that: it decouples policy from software. Your services (or conftest, in your case) hand OPA a JSON document and ask a question; OPA answers using rules written in one dedicated language — Rego.

Rego itself is inspired by Datalog, a decades-old logic query language, extended to work over modern structured documents (JSON/YAML). Two consequences you'll feel immediately:

  • Declarative: you state what must be true, not how to check it. No loops to write; OPA figures out execution.
  • Everything is a document: a Kubernetes manifest, a Dockerfile, a pom.xml — conftest converts them all to JSON, and one language queries them all.
The one-sentence pitch: “Rego lets us write our team's rules once, as code, and check every config against them automatically — instead of rules living in wiki pages and code reviews.”

2 · Your local toolbox

3 · Hello, Rego

Create hello.rego:

package hello                # namespace: reachable as data.hello

greeting := "Hello, Rego!"   # a rule defining a constant value

What you just wrote: a rule

That second line is a rule — the fundamental building block of Rego. A rule is a named definition inside a package: it gives a name (greeting) to a value ("Hello, Rego!"). A policy file is nothing more than a package declaration plus a collection of rules (docs: rules).

greeting defines a constant, the simplest possible rule. Soon your rules will compute values conditionally — “this deployment is denied if it runs as root” — but the concept stays the same: a name, defined in a package, that a query can ask for.

Query it:

$ opa eval --data hello.rego 'data.hello.greeting' --format raw
Hello, Rego!

Now feed it an input document — no policy file needed, a query can read input directly:

$ echo '{"name": "meleu"}' |
  opa eval --stdin-input 'input.name' --format raw
meleu

A rule that validates its input

A constant is the simplest rule; the useful ones have a body — conditions after the keyword if. The rule defines its value only when every condition holds. Create validate.rego:

package validate

valid if input.replicas >= 2   # "valid" exists only when replicas ≥ 2

Now load the rule (--data) and pipe in a document to judge (--stdin-input) — both root trees at once:

$ echo '{"replicas": 3}' |
  opa eval --data validate.rego --stdin-input 'data.validate.valid' --format raw
true

$ echo '{"replicas": 1}' |
  opa eval --data validate.rego --stdin-input 'data.validate.valid' --format raw
      # ← nothing. the rule is *undefined*, not false
When a rule's body fails, the rule produces no value at all — it is undefined. That's why the second run prints nothing (in the default format you'd see {}, an empty result). Undefined is Rego's normal “this rule didn't fire” signal, and it's exactly how conftest policies work: a deny that stays undefined means nothing to complain about. You'll lean on this constantly.

That three-line file is the shape of every config check you'll write: name a condition about input, and let it be defined only when the config is (or isn't) acceptable.

The opa eval options, decoded

OptionShortMeaning
--data hello.rego-dLoad a policy (or data) file. Its contents become part of the data tree — that's why the query says data.hello.greeting.
--input file.json-iProvide the input document from a JSON/YAML file.
--stdin-input-ISame as --input, but read the document from stdin — handy for piping.
--format raw—Print the bare value. Without it, opa eval wraps results in a JSON envelope with query metadata (try it).

One idea hiding in these three lines: package namespaces your rules. Your policy becomes part of a document tree — that's why the query says data.hello.greeting.

The two root documents: input and data

Every value a Rego query can ever see lives in one of exactly two trees:

DocumentWhat it holdsHow it gets there
inputThe one document currently being judged--input / --stdin-input
dataYour policies + any static reference data (lookup tables, allowlists)--data

You just used both: data.hello.greeting came from a file you loaded; input.name came from the document you piped in. Every policy you'll ever write is a question about input, answered by rules living in data (the document model).

Run all three opa eval commands on your machine — Hello, Rego!, meleu, and a valid rule that flips to undefined when you drop replicas to 1. You now have a working policy toolchain and you've written your first validating rule — the same shape that will guard your team's configs.

4 · Check yourself

Answer from memory — one attempt each.

What core problem was OPA created to solve?

OPA decouples policy from software so rules can be written, versioned, and updated independently.

Which document does --stdin-input feed?

input = the document being judged (via --input or --stdin-input); data = policies and files loaded with --data.

In Rego, what is a rule?

Rules are Rego's building block: a name, defined in a package, that a query can ask for. greeting := "Hello, Rego!" is a rule defining a constant.

A rule's if body fails. What value does the rule produce?

A failed body yields undefined, not false. In conftest, an undefined deny means there's nothing to report.

Which command queries a policy file from your terminal?

eval evaluates queries; test runs unit tests; fmt formats code.

5 · Go deeper

Primary source: read the opening sections of the official Policy Language documentation — the single most authoritative Rego reference, and the one to hand coworkers.

Quick lookups: Rego toolbox reference sheet.

Since OPA 1.0 the keyword if is mandatory before a rule's body (its conditions), and contains is mandatory in multi-value rules (rules that build a set). Bodiless constants like greeting := "Hello" need neither. Older blog posts show bodied rules without if — that syntax is dead. Everything in this course uses current syntax.
Ask your teacher (the agent) anything — why := and not =, what happens when you query something that doesn't exist (data.hello.nope), what the JSON envelope fields mean. Follow-up questions are the point.
