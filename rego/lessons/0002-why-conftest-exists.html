Why conftest Exists

Lesson 0002 · ~15 minutes · Mission: conftest policies at work

Every value a Rego query can see lives in input or data (lesson 0001). Keep that in mind — this whole lesson is about who builds input for you.

1 · Same engine, two different questions

conftest is not a rival to OPA — it embeds OPA's Rego engine as a library. Every line of Rego you write for conftest is evaluated by the exact same code as opa eval. What differs is the question each CLI answers:

ToolThe question it answersAnalogy
opa eval“What is the value of this query?”a REPL / calculator
conftest test“Does this file pass our rules?”a test runner (JUnit)

opa is a general-purpose policy engine: it evaluates queries, runs servers, serves any decision any software might ask. conftest is a purpose-built wrapper for exactly one job — testing config files in a terminal or CI pipeline — which happens to be your mission.

2 · The three gaps conftest fills

Gap 1 — Parsers: opa can't even read your files

opa eval --input accepts JSON and YAML. Full stop. Your mission targets Dockerfiles and pom.xml — opa has no idea what those are. conftest ships parsers for 16+ formats (YAML, JSON, XML, Dockerfile, HCL/Terraform, TOML, INI, EDN, …) and converts each file into the input document before your policy ever runs.

conftest's first job is: build input from any config format. Your Rego stays pure — it always queries a JSON document, never “a Dockerfile” or “an XML file.”

Gap 2 — Conventions instead of hand-written queries

With plain opa eval, you decide the package name, write the query string, and interpret the result. conftest replaces all that with fixed conventions:

ConventionDefault
Where policies live./policy/ directory (override: --policy)
Which package is queriedpackage main (override: --namespace)
Rules that mean failuredeny or violation (suffixes allowed: deny_run_as_root)
Rules that mean warningwarn

No query to write. conftest test somefile finds the rules, runs them, prints each message.

Gap 3 — CI ergonomics

A CI pipeline speaks exit codes. conftest translates rule results into them (docs: options):

ResultExit codeWith --fail-on-warn
All rules pass00
Warnings only01
Any failure12

With plain opa eval you'd script this yourself (there is --fail-defined, but you'd be hand-rolling output formatting, multi-file handling, warnings vs failures… i.e. rebuilding conftest).

3 · Feel it: judge a Dockerfile

Create exercises/docker/Dockerfile:

FROM ubuntu:latest
RUN apt-get update
USER root

First, see conftest's parser build the input document — something opa alone cannot do:

$ conftest parse Dockerfile
[
  [
    { "Cmd": "from", "Value": ["ubuntu:latest"], "Stage": 0, ... },
    { "Cmd": "run",  "Value": ["apt-get update"], "Stage": 0, ... },
    { "Cmd": "user", "Value": ["root"], "Stage": 0, ... }
  ]
]

A Dockerfile is now just a JSON document — a list of commands. Now create policy/docker.rego next to it:

package main                          # conftest's default namespace

deny contains msg if {                # syntax dissected next lesson —
    some command in input             # today, focus on the workflow
    command.Cmd == "user"
    command.Value[0] == "root"
    msg := "do not run containers as root"
}

Run it, then check the exit code:

$ conftest test Dockerfile
FAIL - Dockerfile - main - do not run containers as root

1 test, 0 passed, 0 warnings, 1 failure, 0 exceptions

$ echo $?
1

No query written, no parser configured, CI-ready exit code. That is why conftest exists.

conftest parse prints an outer array — one element per parsed document. During conftest test, each document is handed to your policy directly, so input is the command list itself (that's why the rule says some command in input, not input[0]).
Run conftest parse Dockerfile and conftest test Dockerfile on your machine, and confirm echo $? prints 1. Then fix the Dockerfile (change the USER) and watch it pass with exit code 0.

4 · Check yourself

Answer from memory — one attempt each.

conftest embeds OPA's Rego engine unchanged. What does it add on top?

Same engine, same Rego. conftest adds file parsers (16+ formats), fixed conventions (main, deny/warn/violation), and pipeline-friendly exit codes.

By default, which package does conftest test look for rules in?

package main is the default namespace; override with --namespace.

Without extra flags, what exit code does conftest return when a warn rule fires (and no deny)?

Warnings don't fail the run by default — exit 0. Add --fail-on-warn to make warnings exit 1 (failures then exit 2).
