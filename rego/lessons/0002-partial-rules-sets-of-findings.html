Partial Rules: a Set of Findings

One rule, many answers. The rule shape that conftest's deny is built from.

Your win for this lesson By the end you can (1) write a partial set rule that collects zero-or-more findings, (2) explain why an empty result is [] and not undefined, and (3) recognise this as the exact shape of a conftest deny rule.

1. The question changed shape

In Lesson 1, allow answered one yes/no question: a complete rule produces a single value. That's the right tool for "is this allowed?".

But the question you'll actually ask in a pipeline is different: "what's wrong with this config?" The honest answer isn't one boolean — it's a list. Maybe nothing is wrong. Maybe three things are. You want to collect all the problems in one pass, each with its own message. A single value can't hold that; you need a rule that produces a set.

2. The partial set rule

Rego's second rule type builds a document incrementally. The docs call these multi-value rules: each definition, when its body holds, adds one element to a set that all the definitions share. The keyword is contains. Here is a small validator — same generic-JSON spirit as the age check, now reporting problems: 

package signup

# each rule adds ONE message to the `violations` set — when its body holds
violations contains "age must be at least 18" if input.age < 18

# `not input.username` holds when the key is absent (undefined)
violations contains "username is required" if not input.username

# `== ""` holds when the key is present but empty
violations contains "username can't be empty" if input.username == ""

Read it as a definition, like before. There is no loop and no append. You declare, once per problem, "if this body holds, this message belongs in the set." OPA evaluates every definition and unions the results into one document at data.signup.violations. Three definitions, same name — one set.

Notice the username has two definitions, guarding two different failure modes. A missing key and an empty one are not the same thing in Rego: an absent input.username is undefined, and undefined == "" is itself undefined — so == "" alone would silently miss it. not input.username is the presence check that catches the absent case; == "" catches the present-but-empty one. Two problems, two definitions — same pattern as adding any other check.

Kept deliberately simple This validator is a teaching example, not production-ready — it doesn't cover every edge case (a null username, whitespace-only input, type checks, and so on). The goal here is the rule shape; robust presence and value checking is a topic of its own.
contains X
Marks this as a partial (multi-value) rule: X is a candidate element of a set, not the rule's whole value.
if <body>
The same rule body as Lesson 1. If OPA can satisfy it, X is added; if not, this definition contributes nothing.
repeated name
Every rule called violations feeds the same set. Add a check by adding a definition — order doesn't matter.

Two violations definitions both have satisfiable bodies for this input. What is data.signup.violations?

They union. Partial rules with the same name accumulate into one set — that's the whole point. (Compare a complete rule like allow, where two bodies producing different values would conflict.)

3. Run it — you get a set, not a boolean

Save the policy as signup.rego and try an input with two problems:

# bad.json
{ "age": 16, "username": "" }
opa eval -d signup.rego -i bad.json 'data.signup.violations' --format pretty

Output — a set, printed as a JSON array:

[
  "age must be at least 18",
  "username can't be empty"
]

Now feed it a clean input and re-run:

# good.json
{ "age": 21, "username": "meleu" }
[]
The one subtlety that trips everyone When no body holds, the violations key is present in the document, holding the empty set [] — a real, defined value. It is not undefined, and it needs no default. Contrast Lesson 1: a complete rule whose body fails is undefined — its key is simply absent from the document. Partial rules start empty and fill up; complete rules leave no key at all and need a default to guarantee one when they can't fire.

No violation body holds for the given input. What does data.signup.violations evaluate to?

Empty, not undefined. A partial set rule always denotes a set; with no matching bodies that set is simply empty. This is why deny-style rules never need a default — "no findings" is already [].

4. Turn findings back into a verdict

A set of messages is great for humans, but the pipeline wants a decision. Fold the set back into a single boolean with a complete rule — reusing Lesson 1's shape and the built-in count: 

# fail the check if there is at least one violation
deny if count(violations) > 0

Ask for the whole package on the bad input and you see both documents at once:

opa eval -d signup.rego -i bad.json 'data.signup' --format pretty
{
  "deny": true,
  "violations": [
    "age must be at least 18",
    "username can't be empty"
  ]
}

On the clean input, violations is [] and deny is undefined (its body count([]) > 0 can't be satisfied) — so it simply drops out of the output. Same two rule types from two lessons, now working together.

5. Where this shape is heading

You've built a reusable shape A partial set that collects findings, plus a complete rule that folds them into a verdict, is the backbone of almost every real Rego policy you'll write. The key property — "no findings" is already [], a defined value that needs no default — is exactly what makes the pattern pleasant to build on. Policy tooling that runs in a pipeline is built directly on this shape, and we'll get there once the Rego fundamentals are solid — framed, when we do, as just OPA underneath. For now the win is that you can already write and read it. Next lesson: giving those messages real detail (embedding input values with sprintf) and iterating over lists of items with some … in.

Main takeaways
  • A partial (multi-value) set rule — name contains X if <body> — collects zero-or-more elements instead of one value.
  • Multiple definitions with the same name union into one set; add a check by adding a definition.
  • When no body holds, the result is the empty set [] — a defined value, not undefined, needing no default.
  • That's the opposite failure mode from a complete rule, which is undefined when its body can't be satisfied.
  • Fold a set of findings back to a verdict with a complete rule: deny if count(violations) > 0.
  • This "collect findings, fold to a verdict" shape is the backbone of real policies — the exact pattern pipeline policy tooling is built on (covered later).
