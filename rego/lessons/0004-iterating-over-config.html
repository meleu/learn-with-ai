Iterating Over Config

Real manifests are lists. One deny rule, some … in, and a finding that names every offending container.

Your win for this lesson By the end you can (1) use some … in to make one deny rule emit a finding per offending item in a list, (2) build detailed messages with sprintf, (3) flag a missing required field — a container with no readinessProbe — with not, and (4) reach for every when the question is "do all of them pass?".

1. The risk is inside a list

The gate you built in Lesson 3 checked fixed fields — input.spec.replicas, input.metadata.namespace — so one definition per problem was enough. But the checks a platform team really cares about live one level deeper, in a list: the Deployment's containers. Which images are pinned? Which ones run privileged? You don't know how many containers there are, and you want to check each one and name the culprit.

You can't write one definition per container — you don't know them ahead of time. Instead you write one definition and let it iterate: bind a variable to each element in turn, and emit a finding for every element whose body holds.

2. Name the long path once

Conftest gives you the whole parsed file, so a Deployment's containers sit at input.spec.template.spec.containers. Typing that in every rule is noise your coworkers have to re-read each time. A plain helper rule — a complete rule (Lesson 1) whose value is that path — fixes it: 

package main

# name the long path once; every rule below reads better for it
containers := input.spec.template.spec.containers

Conftest only collects rules named deny, warn, and violation — any other rule in the package is a helper, invisible in the output and free for you to use. This one is undefined for a file that isn't a Deployment, and every rule referring to it then quietly skips: the Lesson 1 undefined behaviour, working in your favour again.

3. some … in — fan out over the list

some c in containers introduces a variable c and tells OPA: "try this body for every element of the list." Each binding that satisfies the whole body contributes its element to the result. Pair it with the deny rule you already write and a single definition fans out into many findings: 

# one finding PER container pinned to :latest — `some` iterates the list
deny contains msg if {
	some c in containers
	endswith(c.image, ":latest")
	msg := sprintf("container %q uses the mutable :latest tag", [c.name])
}

The body is still just conditions joined by AND (Lesson 1) — the only new part is that some … in makes c range over the list rather than naming one fixed path. OPA evaluates the body once per element; three containers means three attempts, and each attempt that holds adds one message to the shared deny set.

For that binding, the element contributes a finding. Bindings that fail contribute nothing.
msg :=
:= assigns a local variable inside the body — here, the finding this binding adds.

Point it at a Deployment with three containers:

# deployment.yaml
spec:
  replicas: 3
  template:
    spec:
      containers:
        - name: api
          image: corp/api:1.4.2
          readinessProbe:
            httpGet:
              path: /healthz
              port: 8080
        - name: cache
          image: redis:latest
        - name: web
          image: nginx:latest
          securityContext:
            privileged: true
conftest test deployment.yaml

One definition, two FAIL lines — the two containers on :latest:

FAIL - deployment.yaml - main - container "cache" uses the mutable :latest tag
FAIL - deployment.yaml - main - container "web" uses the mutable :latest tag

2 tests, 0 passed, 0 warnings, 2 failures, 0 exceptions

Three containers, two of them on :latest. Using some c in containers, how many findings does the single deny rule add?

One per match. some runs the body once for each element, and only the bindings whose body holds contribute. Two containers match, so the set has two findings — the non-:latest one simply produces no binding that satisfies the body.

4. sprintf — messages that name the culprit

A finding that says "an image uses :latest" makes a human go hunting. A finding that says which container is instantly actionable — and in a build log, that difference is the whole value of the gate. sprintf(format, args) builds a string from a format template and an array of arguments — one array slot per % verb, in order: 

msg := sprintf("container %q on image %q", [c.name, c.image])
# → container "cache" on image "redis:latest"
VerbFormatsExample → output
%vany value, default form[1,2] → [1, 2]
%sa string, unquotedapi → api
%qa string, quotedapi → "api"
%dan integer18 → 18

The %q quoting is worth the habit: it makes names visibly stand out in a message and survives empty or odd values (an empty name prints as "", not as nothing). Full verb list and more examples live on the Iteration & Strings reference card.

5. More conditions, deeper input

A body can mix iteration with any number of other conditions — still AND. Add a second check for privileged containers, reaching into a nested field: 

# one finding per privileged container
deny contains msg if {
	some c in containers
	c.securityContext.privileged == true
	msg := sprintf("container %q runs privileged", [c.name])
}

This is a second definition of deny, so its findings union into the same set (Lesson 2) and conftest prints them together: 

FAIL - deployment.yaml - main - container "cache" uses the mutable :latest tag
FAIL - deployment.yaml - main - container "web" runs privileged
FAIL - deployment.yaml - main - container "web" uses the mutable :latest tag

3 tests, 0 passed, 0 warnings, 3 failures, 0 exceptions

Notice the safety you get for free: if a container has no securityContext, then c.securityContext.privileged is undefined, the == true comparison is undefined, the body fails for that binding — and it adds nothing. Missing structure quietly skips the check instead of crashing. Here that's exactly what you want (a container without a securityContext genuinely isn't privileged) — unlike Lesson 3's replicas, where the missing key was the problem. Same mechanism; it's on you each time to decide which case you're in.

6. When the absence is the violation

Both checks so far fire on a field that is present and wrong. Plenty of the rules a platform team actually ships are the opposite: the field must simply be there. The canonical one is the readinessProbe — without it Kubernetes considers a pod ready the moment it starts, sends it traffic before the process can serve, and every rollout drops requests (k8s docs — probes). It's a one-line omission with a production-shaped blast radius, which makes it a perfect gate.

You already have every piece: iterate the containers, and keep the ones where the field isn't there. The only new keyword is not: 

# one finding per container missing a readinessProbe
deny contains msg if {
	some c in containers
	not c.readinessProbe
	msg := sprintf("container %q has no readinessProbe", [c.name])
}

not expr holds exactly when expr is undefined (or false). For the api container c.readinessProbe is a value, so not fails and the binding contributes nothing. For cache and web the lookup is undefined — and not flips that into a condition that holds, so the body runs to the end and the finding lands. Same undefined you met one section ago, opposite intent: there it meant "nothing to report", here it is the report. That's Lesson 3's replicas pair again — this time applied per item in a list.

Against the same manifest, all three definitions together:

FAIL - deployment.yaml - main - container "cache" has no readinessProbe
FAIL - deployment.yaml - main - container "cache" uses the mutable :latest tag
FAIL - deployment.yaml - main - container "web" has no readinessProbe
FAIL - deployment.yaml - main - container "web" runs privileged
FAIL - deployment.yaml - main - container "web" uses the mutable :latest tag

5 tests, 0 passed, 0 warnings, 5 failures, 0 exceptions
What this rule does not say not c.readinessProbe asserts presence, nothing more. A container with readinessProbe: {} — an empty, useless probe — is a defined value, so not fails and the gate passes it. Presence checks are cheap to write and easy to satisfy hollowly; when that matters, check the thing you actually need instead, e.g. not c.readinessProbe.httpGet. Deciding how deep to look is the policy author's job, and it's the difference between a gate that changes behaviour and one that teaches people to add empty stanzas.

A container declares readinessProbe: {} and nothing else. What does the rule not c.readinessProbe do with it?

It passes. not only holds for undefined or false, and {} is a perfectly defined value — so the body stops there and adds no finding. To demand a real probe, assert the field you care about, such as not c.readinessProbe.httpGet.

7. every — the "do all of them pass?" question

some asks "is there an element that…?" — perfect for collecting findings. Its mirror image is every, which asks "do all elements satisfy…?" and yields a single boolean. That's a verdict, not a finding, so it belongs in a helper rule that a deny then consumes: 

# helper — true only when EVERY image avoids :latest
default all_pinned := false
all_pinned if {
	every c in containers {
		not endswith(c.image, ":latest")
	}
}

# one whole-manifest finding, instead of one per container
deny contains "every image must be pinned to an immutable tag" if not all_pinned

every c in xs { … } holds when the braced body holds for all bindings; one failing element makes the whole thing fail. As a complete rule it needs a default (Lesson 1) so it answers false rather than going undefined — otherwise not all_pinned would be reasoning about a value that isn't there.

The gotcha everyone meets once every over an empty list is true — there's no element to violate the condition, so "all of them pass" holds vacuously. A Deployment with zero containers reports all_pinned = true and sails through your gate. It's Lesson 3's silent-pass trap wearing a different hat. When "empty" should not pass, add an explicit non-empty check (count(containers) > 0).

containers is the empty list []. What does all_pinned, built with every, evaluate to?

Vacuously true. every fails only when some element breaks the condition; with no elements, none can, so it holds. Guard with a count check when an empty collection should be treated as a failure.

8. A rule body is the AND of its lines

So where does that count guard go? The answer is worth pausing on, because it comes straight from Lesson 1: a rule body is just conditions joined by AND. Every line must hold for the rule to fire — there is no and keyword, the newline is the AND. So to demand "at least one container and all of them pinned", you add the guard as another line in the body, right beside the every: 

default all_pinned := false
all_pinned if {
	count(containers) > 0                  # AND — must be non-empty
	every c in containers {          # AND — and all pinned
		not endswith(c.image, ":latest")
	}
}

On an empty list, count([]) > 0 is false, so the body fails as a whole and the default answers false — the vacuous true can never leak out, and the deny fires as it should. The placement matters: the guard is an outer body line. Tucking it inside the every { … } braces wouldn't help, because that block runs per element — with zero elements it never runs, and the rule is vacuously true again.

Which rules need the guard? Only the "do all pass?" ones. A some-based deny needs no guard: an empty list yields no findings, i.e. [], which already means "nothing wrong". And if an empty manifest should itself be a finding, don't guard all_pinned — add a separate check: deny contains "no containers" if count(containers) == 0. Whether empty is fine or forbidden is a policy decision, not a Rego one.

You want all_pinned to be false for an empty list. Where does count(containers) > 0 belong?

In the body, beside the every. The body is an AND, so the guard joins it as a peer line and the rule needs both. Inside the every braces it would run per element — never, for an empty list — so the rule stays vacuously true; and default takes a bare fallback value, not a condition.

9. Next: collecting values, not just booleans

Where this goes next So far iteration has produced two things: a set of findings (strings) with some + contains, and a single verdict (boolean) with every. The same some can also collect the values themselves — the set of offending container names, a map of name → image — using comprehensions. That's the whole of Lesson 5.

Main takeaways
  • Real config is a list, so checks iterate instead of naming a fixed field.
  • A rule that isn't deny/warn/violation is a helper — invisible to conftest. Use one to name a long path once (containers := input.spec.template.spec.containers).
  • some c in containers runs the body once per element; combined with contains, one definition emits one FAIL line per matching item.
  • sprintf(fmt, [args]) builds detailed messages; %q quotes a string, %s leaves it bare, %d is an integer, %v is any value.
  • A body freely mixes iteration with other conditions (still AND); reaching into a missing nested field is undefined, which safely skips that binding.
  • To require a field, flip that undefined with not: not c.readinessProbe holds precisely when the field is absent — a presence check, which readinessProbe: {} still satisfies.
  • every c in xs { … } answers "do all pass?" as one boolean — and is vacuously true over an empty list, so guard with count when empty should fail.
  • A rule body is the AND of its lines (the newline is the AND); the empty-list guard is an outer body line beside the every, not inside its braces.
  • The same some drives comprehensions like {v | body} that build collections — the subject of Lesson 5.
