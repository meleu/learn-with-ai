Conftest: just OPA underneath

The payoff. Everything you wrote by hand — input, partial rules, sprintf — becomes a real pipeline gate over a real config file, with almost no new Rego.

Your win for this lesson By the end you can (1) point conftest test at a real YAML file and have it parse the file into input for you, (2) recognise deny contains msg if as the Lesson 2 partial rule you already know, and (3) read conftest's PASS/FAIL output and its exit code — the moment a decision becomes enforcement.

1. You have already written conftest policies

Since Lesson 1 you have hand-fed OPA a JSON document with opa eval -i input.json and collected findings into a partial set rule. Conftest is the tool from the mission that does exactly that, for real, in your pipeline. It changes only two things about what you already do:

Change 1
input is your config file, parsed for you — no more hand-written input.json. YAML, JSON, Dockerfile, HCL, TOML and more, picked by file type. (conftest.dev)
Change 2
Rule names are a convention. Conftest collects rules called deny, warn, and violation from package main. That's it — the rule bodies are the Rego you already write.

2. deny is the partial rule you know, renamed

In Lesson 2 you built a set of findings with a contains … if partial rule and called it violations. Conftest wants that exact shape — it just insists on the name deny (or warn/violation) inside package main: 

package main

# a container using the mutable :latest tag → one finding each
deny contains msg if {
	some c in input.spec.containers
	endswith(c.image, ":latest")
	msg := sprintf("%q uses the mutable :latest tag", [c.name])
}

# a container running privileged → one finding each
deny contains msg if {
	some c in input.spec.containers
	c.securityContext.privileged == true
	msg := sprintf("%q runs privileged", [c.name])
}

Look closely: some … in (Lesson 3), sprintf with %q (Lesson 3), two same-named partial rules unioning into one set (Lesson 2), the missing-nested-field-is-undefined safety on securityContext.privileged (Lesson 1). Nothing here is new. Only package main and the name deny are conftest's ask.

Why input.spec.containers and not input.containers? Conftest hands you the whole parsed file as input — so your paths follow the file's real shape. A Kubernetes Pod nests its list under spec.containers, so that's the path. Same mental model as Lesson 1: input is just the document, and you reach into it exactly as it's written.

3. Run it on a real file

Conftest looks in a policy/ directory by default. Put the rule above in policy/deny.rego, and drop a manifest next to it: 

# pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: billing
spec:
  containers:
    - name: api
      image: corp/api:latest
      securityContext:
        privileged: true
    - name: web
      image: nginx:1.27.3
conftest test pod.yaml

The api container trips both rules; web is clean:

FAIL - pod.yaml - main - "api" runs privileged
FAIL - pod.yaml - main - "api" uses the mutable :latest tag

2 tests, 0 passed, 0 warnings, 2 failures, 0 exceptions

Each FAIL line is one element of your deny set — the same finding strings opa eval would have printed, now formatted for a human and a build log. And the piece that makes it a gate: 

echo $?     # → 1  (any deny finding ⇒ non-zero exit)
This is the enforcement point Lesson 1 drew the line between deciding (OPA's job) and enforcing (someone else's). That "someone else" is now concrete: conftest exits non-zero when any deny fires, and your CI step fails the build. The Rego decides; the exit code enforces.

You point conftest test at a Dockerfile instead of YAML. What do you change in the policy?

Only the paths. Conftest parses each format into the same kind of input document; a Dockerfile just has a different shape than a Pod. The rule names (deny), the package (main), and the Rego constructs are identical — you reach into input along whatever structure the parsed file has.

4. deny vs warn vs violation

Conftest recognises three rule names, differing only in how hard they push and, for one, the shape of the message (conftest.dev):

RuleMessageEffect on the build
denya stringfails — exit code 1
warna stringprints a warning, build still passes — exit 0
violationa structured objectsame as deny, but returns rich data, not just text

Reach for warn when something is worth flagging but shouldn't block a merge, and deny when it must. Add a warn rule and rerun, and the tally distinguishes them while the exit code stays 0 if there are no denials: 

3 tests, 2 passed, 1 warning, 0 failures, 0 exceptions

A run reports 0 failures but 2 warnings. What exit code does conftest return?

Zero. Only deny and violation failures make conftest exit non-zero. warn exists precisely to surface an issue without failing the build — the step passes and CI moves on. Use it for advisories, not hard gates.

5. Name your rules so a coworker can read them

Two bare deny blocks work, but the mission is also to teach coworkers. Conftest lets any deny/warn/violation rule carry an underscore suffix — deny_latest_tag, deny_privileged — and still collects it (conftest.dev). The suffix is documentation; conftest ignores it when matching: 

package main

deny_latest_tag contains msg if {
	some c in input.spec.containers
	endswith(c.image, ":latest")
	msg := sprintf("%q uses the mutable :latest tag", [c.name])
}

deny_privileged contains msg if {
	some c in input.spec.containers
	c.securityContext.privileged == true
	msg := sprintf("%q runs privileged", [c.name])
}

Same findings, same output — but now each rule announces what it checks. Prefer one clearly-named rule per concern over one deny with a tangled body.

Main takeaways
  • Conftest is OPA with two conveniences: it parses your config file into input, and it collects rules by name.
  • A deny contains msg if rule is the Lesson 2 partial set rule — just named deny inside package main.
  • input is the whole parsed file, so paths follow its real shape (a Pod's list lives at input.spec.containers).
  • conftest test file.yaml reads policies from policy/ by default; each finding is one FAIL line.
  • deny fails the build (exit 1); warn advises but passes (exit 0); violation is deny with structured data.
  • The exit code is the enforcement point — the decision your Rego computes becomes a CI pass/fail.
  • Suffix rules (deny_privileged) to make each check self-documenting for coworkers.
