Iterating Over Config

Real manifests are lists. One rule, some … in, and a detailed finding for every bad item.

Your win for this lesson By the end you can (1) use some … in to make one partial rule emit a finding per offending item in a list, (2) build detailed messages with sprintf, and (3) reach for every when the question is "do all of them pass?".

1. The config is a list now

In Lesson 2 each check looked at a fixed field — input.age, input.username — so one definition per problem was enough. But the manifests you validate in a pipeline aren't shaped like that. A deployment has a list of containers; a compose file, a list of services. You don't know how many there are, and you want to check each one.

You can't write one definition per container — you don't know them ahead of time. Instead you write one definition and let it iterate: bind a variable to each element in turn, and emit a finding for every element whose body holds.

2. some … in — fan out over the list

some c in input.containers introduces a variable c and tells OPA: "try this body for every element of the list." Each binding that satisfies the whole body contributes its element to the result. Pair it with the contains rule from Lesson 2 and a single definition fans out into many findings. Here's a check that flags any image pinned to the mutable :latest tag: 

package manifest

# one finding PER container pinned to :latest — `some` iterates the list
violations contains msg if {
	some c in input.containers
	endswith(c.image, ":latest")
	msg := sprintf("container %q uses the mutable :latest tag", [c.name])
}

The body is still just conditions joined by AND (Lesson 1) — the only new part is that some … in makes c range over the list rather than naming one fixed path. OPA evaluates the body once per element; three containers means three attempts, and each attempt that holds adds one message to the shared violations set.

For that binding, the element contributes a finding. Bindings that fail contribute nothing.

Save it as manifest.rego and feed it a list with three containers:

# input.json — real manifests are arrays
{
  "containers": [
    { "name": "api",   "image": "corp/api:1.4.2" },
    { "name": "cache", "image": "redis:latest" },
    { "name": "web",   "image": "nginx:latest" }
  ]
}
opa eval -d manifest.rego -i input.json 'data.manifest.violations' --format pretty

One definition, two findings — the two containers on :latest:

[
  "container \"cache\" uses the mutable :latest tag",
  "container \"web\" uses the mutable :latest tag"
]

Three containers, two of them on :latest. Using some c in input.containers, how many findings does the single violations rule add?

One per match. some runs the body once for each element, and only the bindings whose body holds contribute. Two containers match, so the set has two findings — the non-:latest one simply produces no binding that satisfies the body.

3. sprintf — messages that name the culprit

A finding that says "an image uses :latest" makes a human go hunting. A finding that says which container is instantly actionable. sprintf(format, args) builds a string from a format template and an array of arguments — one array slot per % verb, in order: 

msg := sprintf("container %q on image %q", [c.name, c.image])
# → container "cache" on image "redis:latest"
VerbFormatsExample → output
%vany value, default form[1,2] → [1, 2]
%sa string, unquotedapi → api
%qa string, quotedapi → "api"
%dan integer18 → 18

The %q quoting is worth the habit: it makes names visibly stand out in a message and survives empty or odd values (an empty name prints as "", not as nothing). Full verb list and more examples live on the new Iteration & Strings reference card.

4. More conditions, deeper input

A body can mix iteration with any number of other conditions — still AND. Add a second check for privileged containers, reaching into a nested field: 

# one finding per privileged container
violations contains msg if {
	some c in input.containers
	c.securityContext.privileged == true
	msg := sprintf("container %q runs privileged", [c.name])
}

This is a second definition of violations, so its findings union into the same set (Lesson 2). Notice the safety you get for free: if a container has no securityContext, then c.securityContext.privileged is undefined, the == true comparison is undefined, the body fails for that binding — and it simply adds nothing. Missing structure quietly skips the check instead of crashing. That's the same undefined from Lesson 1, now working in your favour.

5. every — the "do all of them pass?" question

some asks "is there an element that…?" — perfect for collecting findings. Its mirror image is every, which asks "do all elements satisfy…?" and yields a single boolean. Use it for a whole-manifest verdict: 

# true only when EVERY image avoids :latest
default all_pinned := false
all_pinned if {
	every c in input.containers {
		not endswith(c.image, ":latest")
	}
}

every c in xs { … } holds when the braced body holds for all bindings; one failing element makes the whole thing fail. As a complete rule it needs a default (Lesson 1) so it answers false rather than going undefined.

The gotcha everyone meets once every over an empty list is true — there's no element to violate the condition, so "all of them pass" holds vacuously. A manifest with zero containers reports all_pinned = true. When "empty" should not pass, add an explicit non-empty check (count(input.containers) > 0).

input.containers is the empty list []. What does all_pinned, built with every, evaluate to?

Vacuously true. every fails only when some element breaks the condition; with no elements, none can, so it holds. Guard with a count check when an empty collection should be treated as a failure.

6. A rule body is the AND of its lines

So where does that count guard go? The answer is worth pausing on, because it comes straight from Lesson 1: a rule body is just conditions joined by AND. Every line must hold for the rule to fire — there is no and keyword, the newline is the AND. So to demand "at least one container and all of them pinned", you add the guard as another line in the body, right beside the every: 

default all_pinned := false
all_pinned if {
	count(input.containers) > 0        # AND — must be non-empty
	every c in input.containers {   # AND — and all pinned
		not endswith(c.image, ":latest")
	}
}

On an empty list, count([]) > 0 is false, so the body fails as a whole and the default answers false — the vacuous true can never leak out. The placement matters: the guard is an outer body line. Tucking it inside the every { … } braces wouldn't help, because that block runs per element — with zero elements it never runs, and the rule is vacuously true again.

Which rules need the guard? Only the "do all pass?" ones. A some/violations rule needs no guard: an empty list yields no findings, i.e. [], which already means "nothing wrong". And if an empty manifest should itself be a finding, don't guard all_pinned — add a separate check: violations contains "no containers" if count(input.containers) == 0. Whether empty is fine or forbidden is a policy decision, not a Rego one.

You want all_pinned to be false for an empty list. Where does count(input.containers) > 0 belong?

In the body, beside the every. The body is an AND, so the guard joins it as a peer line and the rule needs both. Inside the every braces it would run per element — never, for an empty list — so the rule stays vacuously true; and default takes a bare fallback value, not a condition.

7. Next: collecting values, not just booleans

Where this goes next So far iteration has produced two things: a set of findings (strings) with some + contains, and a single verdict (boolean) with every. The same some can also collect the values themselves — the set of offending container names, a map of name → image — using comprehensions. That's the whole of Lesson 4.

Main takeaways
  • Real config is a list, so checks iterate instead of naming a fixed field.
  • some c in input.containers runs the body once per element; combined with contains, one definition emits one finding per matching item.
  • sprintf(fmt, [args]) builds detailed messages; %q quotes a string, %s leaves it bare, %d is an integer, %v is any value.
  • A body freely mixes iteration with other conditions (still AND); reaching into a missing nested field is undefined, which safely skips that binding.
  • every c in xs { … } answers "do all pass?" as one boolean — and is vacuously true over an empty list, so guard with count when empty should fail.
  • A rule body is the AND of its lines (the newline is the AND); the empty-list guard is an outer body line beside the every, not inside its braces.
  • The same some drives comprehensions like {v | body} that build collections — the subject of Lesson 4.
